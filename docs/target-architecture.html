1. Where we are today

After Phases 1–6, openctl is a single-binary controller with persisted state, async operations, an in-process provider model (proxmox + k3s compiled in), declarative apply with destructive guardrails, drift detection on Get, and a one-command macOS install. It's a working homelab tool today.

But the execution model is still largely procedural. k3s.Apply imperatively walks through "create VMs, install k3s, install agents, save state" as a single block of code. The dispatcher serializes per-resource ops with a parent_id column reserved for parent/child relationships, but it doesn't yet use that column meaningfully. Drift is observed but doesn't trigger rebuilds at the sub-resource level. Scaling an existing cluster up isn't supported because there's no way to express "add a new node task that depends on the existing first-CP's join token."

The pieces foreshadow a build graph; they don't yet form one.

3. In Build Systems à la Carte terms

Mokhov, Mitchell, and Peyton Jones decompose build systems into three orthogonal dimensions: persistent build info, scheduler, and rebuilder. The target architecture lands at a specific, defensible point in their design space.

Persistent build info: verifying trace

We store (resource_key → input_hash, refs_hash, output_summary, last_succeeded_at) after each successful task. Not a full constructive trace (which would cache outputs and let us serve from cache without re-running) — infrastructure operations don't have pure-function semantics, so caching outputs doesn't help; we always have to read reality. But the hash-of-inputs is enough to answer "do I need to act?" which is the actual question.

Scheduler: suspending

Tasks may depend on values that aren't materialized at plan time. The k3s join token isn't known until cp-0 finishes installing. A suspending scheduler lets the task announce "I need K3sNode/cp-0's status.nodeToken" and the scheduler resolves it then — running cp-0's install first if needed.

Topological scheduling falls out as the common case when all refs are statically inspectable from the manifest. Restarting scheduling — the Excel-style "if it changed mid-run, redo everything that read it" — we explicitly don't want; too expensive for infrastructure.

Rebuilder: trace-verifying

Compare current (input_hash, refs_hash) to last success. If match → skip. If mismatch → run, update trace. This is what makes drift convergence work end-to-end, and what makes scale-up cheap (existing tasks are skipped because their hashes haven't changed).

Comparison

In BSALC's taxonomy table, that puts us in roughly the same cell as Shake — suspending scheduler + verifying traces. Bazel is constructive-trace + restarting; Make is dirty-bit + topological. We're closer to Shake than to either, which feels right because infrastructure has the same "deps known at run-time, outputs not cacheable" character as build systems that touch generated headers.

4. Compared to Crossplane

Crossplane is genuinely the closest existing thing. Both have k8s-shaped resources, providers that own CRUD for a resource family, a composer pattern (Crossplane's Composition + XR; ours: Cluster planning child manifests), drift detection through periodic Read, typed schemas validated on submit, and conditions in status.

The "Crossplane without k8s" framing is exactly right. We get the k8s model (declarative resources, providers, compositions, conditions) without the k8s runtime. The tradeoff: less mature ecosystem, but radically cheaper to run, no chicken-and-egg bootstrap problem, no etcd, no operators. A reasonable choice for a tool that wants to manage a small fleet, not a multi-tenant cloud.

5. New primitives

Three additions to the wire protocol enable everything downstream. None are huge on their own; together they're what makes the DAG model work.

5.1 Typed resource references

The current protocol can't express "this field's value comes from another resource." That's needed for the join-token problem (the worker's K3sNode spec needs cp-0's nodeToken) and any future cross-resource value flow.

Two designs to choose from. Design A: typed references on the wire. The CUE schema accepts either a literal or a #Ref object:

K3sNodeSpec: {
    join?: {
        serverIP: string | #Ref   // either literal IP or a ref
        token:    string | #Ref
    }
    ...
}

#Ref: {
    ref: {
        kind:  string
        name:  string
        field: string  // e.g. "status.nodeIP"
    }
}

Wire form (YAML):

spec:
  join:
    serverIP:
      ref: { kind: K3sNode, name: dev-cp-0, field: status.nodeIP }
    token:
      ref: { kind: K3sNode, name: dev-cp-0, field: status.nodeToken }

Design B: string templating — the Terraform answer:

spec:
  join:
    serverIP: "${ref:K3sNode/dev-cp-0/status.nodeIP}"
    token:    "${ref:K3sNode/dev-cp-0/status.nodeToken}"

Cheaper to add, reads naturally, but you pay the standard template-language taxes: parse errors deferred to apply time, no static typing, easy to typo. Recommend Design A — openctl already validates with CUE, and the disjunction-of-types pattern is what CUE is good at.

The resolution happens in the dispatcher, before provider.Apply is called:

// In dispatcher.execute, before calling p.Apply(manifest):
resolved, refs, err := refs.Resolve(ctx, manifest, d.registry)
if err != nil {
    _ = d.store.Complete(ctx, op.ID, StatusFailed,
        fmt.Sprintf("resolve refs: %v", err), "")
    return
}
// `refs` is the list of (kind, name) the manifest depended on — record
// them on the op so future drift/rebuild can ask "which deps did this
// op actually read?". Mirrors Shake/Nix's verifying-trace.
result, err := p.Apply(ctx, resolved)

Capturing the set of refs an apply actually resolved gives you the input to a verifying trace: store (task_key, manifest_hash, refs_hash) and the rebuilder can ask "are any of the resolved values different from last time?" — that's how you correctly trigger a worker reinstall when only the join token changed.

5.2 Conditions on status

Borrow verbatim from Kubernetes. Small wire addition, big semantic win:

status:
  installedVersion: v1.29.0+k3s1
  serviceState: failed
  inputHash: a7b3c9...
  conditions:
    - type: Healthy
      status: "False"
      reason: PartialInstall
      message: "service files present at /etc/systemd/system/k3s.service but systemctl is-active returns failed"
      lastTransitionTime: 2026-05-25T14:22:11Z
    - type: Reachable
      status: "True"
      lastTransitionTime: 2026-05-25T14:22:09Z

Three condition types per resource, echoing k8s patterns:

TypeTrue meansSet by
ReachableSSH / network connectivity worksGet
Healthyservice is active AND installedVersion matches inputHash's versionGet
Progressingan Apply is currently running on this resourcedispatcher

The lastTransitionTime is the audit trail — even when everything's fine now, the history of state changes is visible via openctl ctl get.

5.3 Recovery policies

Conditions tell you what is wrong; a per-resource recovery policy tells the controller what to do about it.

spec:
  recovery: AutoRepair   # AutoRepair | Manual | RecreateOnly
AutoRepair
(default, the k8s controller-style answer): on Apply, if Healthy=False and Reason=PartialInstall (or similar repairable reasons), do delete-then-create. Print a line: "K3sNode dev-cp-0: PartialInstall detected, auto-repairing (delete + recreate)."
Manual
Apply refuses with FailedPrecondition: Healthy=False, Reason=PartialInstall; remediate manually or set recovery: AutoRepair. The user gets full visibility via get, decides themselves.
RecreateOnly
Forces delete-then-create on every Apply where Healthy=False, no matter the reason. The "I don't trust in-place updates" knob.

6. Concrete sketch: K3sNode as a first-class resource

The first imperative-thing-made-declarative is k3s installation per node. Today it lives in 200 lines of imperative procedure inside cluster.InstallK3s. Pulled out as a resource:

apiVersion: k3s.openctl.io/v1
kind: K3sNode
metadata:
  name: dev-cp-0
spec:
  role: server                          # server | agent
  node:
    ip: 192.168.0.233
    ssh:
      user: ubuntu
      privateKeyPath: ~/.ssh/id_ed25519
  k3s:
    version: v1.29.0+k3s1
    extraArgs: [--disable=traefik]
  join:                                 # omitted for the first server
    serverIP: 192.168.0.233             # resolved from sibling K3sNode at plan time
    token: <opaque>                     # resolved from sibling K3sNode's status
  recovery: AutoRepair
status:
  installedVersion: v1.29.0+k3s1
  serviceState: active
  nodeToken: <opaque>                   # only populated for role=server
  inputHash: a7b3c9...
  conditions: [...]

Provider CRUD (sketch)

func (p *Provider) Apply(ctx context.Context, m *protocol.Resource) (*protocol.Resource, error) {
    spec, _ := resources.ParseK3sNodeSpec(m)
    observed, _ := p.Get(ctx, "K3sNode", m.Metadata.Name)

    healthy := isConditionTrue(observed, "Healthy")
    hashMatches := observed != nil && observed.Status["inputHash"] == spec.inputHash()

    switch {
    case observed == nil:
        return p.create(ctx, spec)

    case healthy && hashMatches:
        return observed, nil  // converged

    case !healthy:
        switch spec.Recovery {
        case "Manual":
            return nil, fmt.Errorf("not healthy (%s); recovery=Manual, refusing auto-repair",
                conditionReason(observed, "Healthy"))
        case "AutoRepair", "RecreateOnly", "":
            log.Printf("K3sNode %s: %s — auto-repairing",
                m.Metadata.Name, conditionReason(observed, "Healthy"))
            if err := p.Delete(ctx, "K3sNode", m.Metadata.Name); err != nil {
                return nil, fmt.Errorf("repair-delete: %w", err)
            }
            return p.create(ctx, spec)
        }

    case !hashMatches:
        return p.update(ctx, spec, observed)  // inputs changed, deliberate update
    }
    return observed, nil
}

Reality is read first (the Terraform central trick). The plan is derived from comparing observed to desired. The imperative side-effect (create runs the SSH installer) only fires when the plan calls for it.

The input hash:

// What we hash for the re-run decision. Everything that genuinely affects
// what's installed/configured goes in; connection material doesn't.
func (s *K3sNodeSpec) inputHash() string {
    h := sha256.New()
    fmt.Fprintf(h, "role=%s\n", s.Role)
    fmt.Fprintf(h, "version=%s\n", s.K3s.Version)
    fmt.Fprintf(h, "args=%s\n", strings.Join(sortedCopy(s.K3s.ExtraArgs), ","))
    if s.Join != nil {
        fmt.Fprintf(h, "joinServer=%s\n", s.Join.ServerIP)
        // Note: token is NOT in the hash — rotating it shouldn't force reinstall.
    }
    return hex.EncodeToString(h.Sum(nil))
}

Cluster as a composer

The existing Cluster provider stops doing the work directly. Its Apply becomes a planner that submits child resources through the same controller pipeline:

func (p *ClusterProvider) Apply(ctx context.Context, m *protocol.Resource) (*protocol.Resource, error) {
    spec, _ := resources.ParseClusterSpec(m)
    cpNames, workerNames := resources.NodeNames(m.Metadata.Name, spec)

    // VMs first (proxmox provider, independent of each other).
    for _, vmName := range append(cpNames, workerNames...) {
        p.applyChild(ctx, vmManifestFor(vmName, spec))
    }

    // First server bootstraps; no join.
    firstCP := cpNames[0]
    firstNodeRes, _ := p.applyChild(ctx, k3sNodeManifest(firstCP, "server", spec, nil))

    // Token is a "computed attribute" dep — Terraform makes you declare these
    // explicitly. Here, the scheduler resolved a ResourceRef pointing at
    // firstNodeRes.Status["nodeToken"].
    join := &JoinConfig{
        ServerIP: firstNodeRes.Status["nodeIP"].(string),
        Token:    firstNodeRes.Status["nodeToken"].(string),
    }

    // Remaining servers + workers all join the first.
    for _, cp := range cpNames[1:] {
        p.applyChild(ctx, k3sNodeManifest(cp, "server", spec, join))
    }
    for _, w := range workerNames {
        p.applyChild(ctx, k3sNodeManifest(w, "agent", spec, join))
    }

    // Per-node openctl-k3s-agent install — separate resource, separate cadence.
    for _, n := range append(cpNames, workerNames...) {
        p.applyChild(ctx, agentInstallManifest(n, spec))
    }

    return p.assembleClusterStatus(ctx, m, spec, cpNames, workerNames)
}

Each applyChild is a Submit to the operations store with parent_id set. Phase 3 reserved that column for exactly this. Now those parent-child ops mean something real — a build graph, not just logical grouping.

What you get vs. what stays painful

Wins

  • Drift on a per-node basis. openctl ctl get K3sNode dev-cp-0 shows current version + serviceState + inputHash. Bump k3s.version in the cluster manifest, only K3sNode resources see drift; VMs untouched.
  • Scale-up falls out. Adding a worker = new VirtualMachine + new K3sNode(agent) tasks. Scheduler sees firstCP K3sNode is already succeeded, doesn't re-run. Just installs on the new node and joins.
  • Version bumps are cheap. Change k3s.version → only K3sNode tasks see input hash change → they update (reinstall) sequentially or in parallel. VMs untouched.
  • The observability agent decouples. Bumping openctl-k3s-agent doesn't touch k3s. Two separate input hashes, two separate convergence cycles.

Stays painful

  • First-server-vs-join ordering. Always sequential. The DAG correctly serializes it, but the wall-clock floor doesn't move — minimum apply time is bounded by the longest path through the graph.
  • "Update" semantics for k3s version bumps in place are k3s-version-dependent and not always safe. Honest v1 is destroy+recreate per node (rolling) — which means brief outage windows even for "minor" bumps.
  • Naming. Using IP as a resource name is brittle (DHCP, recreation). Want a stable logical name (dev-cp-0) + a separate IP lookup. Small registry addition.

7. Roadmap: Phases 7–10

We've shipped Phases 1–6. The architectural goal is reachable in three more phases plus an optional one. Each is shippable on its own, and the intermediate states are coherent.

Phase 7: Refs + Conditions + Plantarget

Goal: add the wire-level vocabulary the DAG model needs, without yet using it for execution.

  • Add ResourceRef to the protocol (Design A — CUE disjunction, typed)
  • Add conditions[] to Resource.Status, Kubernetes-shaped
  • Add recovery policy field to spec
  • Add Provider.Plan(manifest) → []ChildManifest as an optional interface; default = "single task = current Apply"
  • Scheduler resolves refs before calling provider; captures resolved deps on the operation row

Outcome: typed cross-resource references and conditions are available, even though no provider uses Plan yet. Existing providers keep working unchanged.

Phase 8: K3sNode + decompose Clustertarget

Goal: prove the model with the most-painful imperative path. Cluster becomes a composer.

  • Implement K3sNode resource + provider (per the sketch above)
  • Implement AgentInstall as a sibling resource
  • Refactor Cluster.Apply to Cluster.Plan returning K3sNode + AgentInstall + VM child manifests with explicit refs
  • Now the cluster apply naturally produces a DAG, executed via existing parent_id ops

Outcome: scale-up problem evaporates — it's just "compose a manifest with more workers and apply." The Phase 5.x followup is no longer a special case.

Phase 9: Verifying-trace rebuildertarget

Goal: close the loop on declarative convergence. The rebuilder skips work intelligently.

  • Add task_results table: (resource_key, input_hash, refs_hash, output_summary, last_succeeded_at)
  • Apply path: before calling provider, check if (input, refs) hash matches; if yes and Healthy=True, skip with status succeeded (cached)
  • Version bumps, extraArgs changes, in-place spec changes all work declaratively

Outcome: drift triggers correct rebuilds. The "change one node's args, re-apply, only that node reinstalls" workflow works end to end.

Phase 10 (optional, much later): Continuous reconciletarget

Goal: Crossplane-style continuous loop, if you want it.

  • Background ticker re-Gets each managed resource on an interval
  • If drift detected and recovery: AutoRepair, enqueue a reconcile op
  • Per-resource reconcile.interval spec field, throttling, jitter

Outcome: brings us to "always reconciling" mode. The architecture supports it without forcing it.

Sequencing rationale

Phase 7 is the smallest and unlocks the most — it's the language to talk about deps. Phase 8 is the proof point — Cluster as composer actually working against real infrastructure. Phase 9 is what makes drift mean something at apply time. Phase 10 is the optional "always reconciling" mode.

8. Open design questions

Surfaced from the K3sNode sketch and the broader architecture, in roughly the order they need to be decided:

  1. Where does inputHash live — on the node (marker file) or in controller state?

    Marker file is more robust to controller state loss (re-install controller, state intact). Controller-side is faster (no SSH for Get) and cleaner; you accept that controller state loss = "everything looks like drift."

    Provisional answer: on the node. Resilience wins for a homelab tool.

  2. Is K3sNode one resource type with a role field, or two (K3sServer + K3sAgent)?

    One type with role-as-data, following aws_instance's lead — most lifecycle code overlaps; role is just a discriminator.

    Provisional answer: one type.

  3. How does the planner pass the join token from first-CP's status into the agent K3sNode manifests?

    Either: (a) synchronously Apply first-CP, read its status, then submit the rest as already-resolved manifests; or (b) use a ResourceRef placeholder that the scheduler resolves at execution time. (b) is the proper DAG-shaped answer and is what Phase 7's ResourceRef protocol delivers.

    Provisional answer: ResourceRef. The whole point of Phase 7.

  4. Recovery on a partially-installed node?

    Report status to user via conditions[]; act based on recovery spec field. Default AutoRepair = delete-then-create. Like Kubernetes: visibility + policy + a default that does the right thing.

    Resolved.

  5. Should refs be resolvable across providers?

    e.g. can a K3sNode's spec ref a VirtualMachine's status.ip? Yes — the registry already routes by apiVersion. Worth being explicit about it in the Phase 7 spec so the resolver doesn't accidentally scope to single-provider.

  6. What's the parallelism cap?

    The topological scheduler can run N independent tasks at once. Sensible default for a homelab is probably 4–8; configurable per-provider (proxmox API may rate-limit; SSH may not). Worth a flag.
